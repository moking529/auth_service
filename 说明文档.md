## 项目规划

### 项目概述
开发基于Django + Django REST Framework的用户认证和授权服务，实现用户注册、登录、JWT认证、权限控制等功能。

### 功能概述
- 用户注册、登录和JWT认证
- 细粒度权限管理（基于资源类型和操作类型）
- 权限组管理
- API权限检查服务
- 用户配置文件管理
- 日志记录系统

### 技术栈
- **后端框架**：Django 5.2.7、Django REST Framework 3.14+
- **认证授权**：JWT (JSON Web Token)，access_token有效期2小时、refresh_token有效期7天
- **数据库**：开发环境和测试环境使用SQLite，生产环境使用PostgreSQL
- **缓存系统**：Redis（已启用，CACHE_ENABLED = True）
- **后端语言**：Python 3.9-3.14.0
- **部署方式**：Python虚拟环境部署（支持Linux和Windows操作系统）

- 测试阶段：1天

## 实施方案

### 核心功能
1. **用户注册**
   - 支持输入用户名、密码、手机号（可选）、角色（默认普通用户，可选管理员）
   - 密码自动加密存储

2. **用户登录**
   - 输入用户名 + 密码
   - 返回JWT令牌（access_token有效期2小时，refresh_token有效期7天）

3. **令牌刷新**
   - 使用refresh_token获取新的access_token

4. **用户信息获取**
   - 登录后查询自身信息
   - 令牌有效性验证

5. **权限控制**
   - 普通用户：只能查看自己的信息
   - 管理员：可以查看所有用户列表

6. **静态文件服务**
   - 支持静态文件收集和访问
   - 配置优化的缓存策略

## 宝塔面板部署补充说明

### 项目配置

1. 在项目根目录创建`.env`文件：
   ```
   # 基本配置
   DEBUG=False
   SECRET_KEY=your_secret_key_here  # 生成一个随机的密钥
   ALLOWED_HOSTS=your-domain.com,www.your-domain.com
   
   # 数据库配置
   DATABASE_URL=postgres://auth_service_user:your_password@localhost:5432/auth_service_db
   # 或MySQL配置
   # DATABASE_URL=mysql://auth_service_user:your_password@localhost:3306/auth_service_db
   
   # Redis配置
   REDIS_CONFIG_HOST=localhost
   REDIS_CONFIG_PORT=6379
   REDIS_CONFIG_PASSWORD=your_redis_password
   REDIS_CONFIG_DB=0
   CACHE_ENABLED=True
   
   # JWT配置
   ACCESS_TOKEN_LIFETIME=2  # 小时
   REFRESH_TOKEN_LIFETIME=7  # 天
   ```

2. **静态文件配置**：确保`auth_service/settings.py`文件中包含以下静态文件配置：
   ```python
   # Static files (CSS, JavaScript, Images)
   # https://docs.djangoproject.com/en/5.2/howto/static-files/
   
   STATIC_URL = '/static/'
   STATIC_ROOT = os.path.join(BASE_DIR, 'staticfiles')
   
   # 静态文件目录列表
   # 只添加存在的静态文件目录，避免警告
   STATICFILES_DIRS = []
   static_dir = BASE_DIR / 'static'
   if static_dir.exists():
       STATICFILES_DIRS.append(str(static_dir))
   
   # 静态文件查找器
   STATICFILES_FINDERS = [
       'django.contrib.staticfiles.finders.FileSystemFinder',
       'django.contrib.staticfiles.finders.AppDirectoriesFinder',
   ]
   ```

3. **收集静态文件**：
   - 在终端中输入：`python manage.py collectstatic --noinput`
   - 这将把所有静态文件收集到`staticfiles`目录中
   - **注意事项**：
     - 项目使用了智能配置，只在静态文件目录实际存在时才将其添加到STATICFILES_DIRS，避免出现"The directory does not exist"警告
     - 在部署到生产环境前，必须执行此命令
     - 当更新了Django版本或安装了新的依赖后，需要重新执行此命令
     - 如果需要强制重新收集所有文件（覆盖现有文件），使用：`python manage.py collectstatic --clear --noinput`
   - 如果需要添加自定义静态文件，请确保在项目根目录创建`static`目录并设置适当的权限

### 配置反向代理和静态文件服务

1. 配置反向代理：
   - 在刚创建的网站右侧，点击`设置`
   - 选择`反向代理`，点击`添加反向代理`
   - 填写以下信息：
     - 代理名称：`auth_service`
     - 目标URL：`http://127.0.0.1:8000`（与Python项目管理器中设置的端口一致）
     - 发送域名：选择`$host`
     - 启用Websocket：勾选
   - 点击`保存`

2. 配置静态文件服务（重要）：
   - 在网站设置中，点击`配置文件`
   - **确保Nginx配置文件顶部已包含MIME类型设置**：`include mime.types;`
   - 在`location /`块之前添加以下静态文件配置：
     ```nginx
     # 确保包含MIME类型设置（必须在配置文件顶部）
     include mime.types;
     default_type application/octet-stream;
     
     # 静态文件配置（必须在location / 块之前）
     location /static/ {
         alias /www/wwwroot/auth_service/staticfiles/;
         expires 30d;
         access_log off;
         # 确保正确设置MIME类型处理
         types {
             text/css css;
             application/javascript js;
             image/png png;
             image/jpeg jpg jpeg;
             image/gif gif;
             font/woff woff;
             font/woff2 woff2;
             font/ttf ttf;
         }
         # 允许所有访问，避免认证问题
         allow all;
     }
     ```
   - 点击`保存`并重启Nginx服务
   - **注意**：静态文件配置必须在`location /`块之前，否则可能被反向代理配置覆盖

### 常见问题解决

**静态文件无法访问或出现MIME类型错误**
- **症状**：浏览器控制台显示"Refused to apply style from URL because its MIME type ('text/html') is not a supported stylesheet MIME type"或404错误
- **原因分析**：
  1. 静态文件路径配置错误，导致请求返回HTML错误页面
  2. Nginx MIME类型配置缺失或不正确
  3. 静态文件目录权限不足
  4. collectstatic命令未正确执行
  5. 开发/生产环境配置差异导致的问题

- **详细排查步骤**：
  1. **验证静态文件是否存在**：
     - 检查服务器上`/www/wwwroot/auth_service/staticfiles/`目录是否存在
     - 确认`staticfiles/admin/css/`目录下包含必要的CSS文件（base.css, login.css等）
     - 运行`ls -la /www/wwwroot/auth_service/staticfiles/`查看文件权限
  
  2. **重新执行collectstatic**：
     - 在项目目录下运行：`python manage.py collectstatic --noinput`
     - 确认看到"163 unmodified"或类似成功信息
  
  3. **检查Nginx配置（最关键）**：
     - 确认`include mime.types;`在配置文件顶部
     - 确认静态文件配置块`location /static/`在`location /`块之前
     - 验证`alias`路径与实际的`staticfiles`目录路径完全匹配
     - 检查是否有其他location块可能拦截了/static/请求
  
  4. **检查文件权限**：
     - 设置静态文件目录权限：`chmod -R 755 /www/wwwroot/auth_service/staticfiles/`
     - 确保Nginx用户（通常是www或nginx）有权限访问静态文件
  
  5. **开发环境与生产环境差异**：
     - 开发环境（DEBUG=True）：Django自动处理静态文件服务
     - 生产环境（DEBUG=False）：必须通过Nginx等Web服务器提供静态文件
     - 确保在生产环境中正确配置了ALLOWED_HOSTS

- **解决方案**：
  1. **修复Nginx配置**（最常见解决方案）：
     ```nginx
     # 确保在配置文件顶部
     include mime.types;
     default_type application/octet-stream;
     
     # 静态文件配置必须在location / 块之前
     location /static/ {
         alias /www/wwwroot/auth_service/staticfiles/;
         expires 30d;
         access_log off;
         # 明确指定常见MIME类型
         types {
             text/css css;
             application/javascript js;
             image/png png;
             image/jpeg jpg jpeg;
             image/gif gif;
             font/woff woff;
             font/woff2 woff2;
             font/ttf ttf;
         }
         # 确保允许访问
         allow all;
     }
     ```
  
  2. **重新收集静态文件**：
     ```bash
     python manage.py collectstatic --clear --noinput
     ```
  
  3. **修复路径问题**：
     - 如果服务器上的项目路径与配置中的路径不同，更新Nginx配置中的alias路径
     - 确保路径末尾没有多余的斜杠
  
  4. **权限修复**：
     ```bash
     chown -R www:www /www/wwwroot/auth_service/staticfiles/
     chmod -R 755 /www/wwwroot/auth_service/staticfiles/
     ```
  
  5. **重启Nginx和应用服务**：
     ```bash
     service nginx restart
     # 重启Django应用服务
     ```
  
  6. **测试静态文件URL**：
     - 使用curl测试：`curl -I http://your-domain.com/static/admin/css/base.css`
     - 确认返回正确的Content-Type: text/css
